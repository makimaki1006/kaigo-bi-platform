```

- Status__c LIKE '%商談中%'
- Status__c LIKE '%プロジェクト進行中%'
- Status__c LIKE '%深耕対象%'
```

- 除外対象: `data/output/contract_accounts_*.csv` 内の電話番号

## 処理フロー（全体像）

```
[STEP 1] データ準備
├─ スクレイピングデータ読み込み（媒体別）
├─ 成約先電話番号読み込み
├─ 電話済みリスト読み込み
└─ Salesforce既存データ読み込み（Lead/Account/Contact）

[STEP 2] 電話番号正規化・抽出
├─ 数字のみ抽出（10〜11桁）
├─ 固定電話/携帯電話を分類（070/080/090判定）
└─ 重複除去

[STEP 3] 除外処理
├─ 成約先電話番号一致 → 除外
└─ 電話済み電話番号一致 → 除外

[STEP 4] Salesforce突合
├─ 電話番号で既存レコード検索
├─ マッチあり → 更新対象（Lead/Account/Contact）
└─ マッチなし → 新規リード候補

[STEP 5] セグメント分析
├─ 代表者直通判定（担当者名 = 代表者名）
├─ 携帯電話有無判定
└─ 最強セグメント特定（代表者直通 × 携帯あり）

[STEP 6] CSV生成
├─ 新規リード作成用CSV
├─ Lead更新用CSV
├─ Account更新用CSV
└─ 除外リスト

[STEP 7] 所有者割り当て
├─ 最強セグメント → 市来
├─ 人材開発チーム → 7名均等
├─ 深堀チーム → チーム単位
└─ 服部チーム → チーム単位

[STEP 8] バックアップ取得

[STEP 9] Salesforceインポート（ユーザー確認後）
├─ 新規リード作成
├─ 既存Lead更新
└─ 既存Account更新

[STEP 10] メモ欄差別化
├─ 新規: 【新規作成】有料媒体突合 YYYY-MM-DD
└─ 既存: 【既存更新】有料媒体突合 YYYY-MM-DD

[STEP 11] レポート作成
├─ 新規作成リード用
├─ 既存更新リード用
└─ 既存更新Account用
```

## セグメント定義

### セグメント分類

| セグメント | 定義 | 優先度 |
|-----------|------|--------|
| **代表者直通** | 担当者名 = 代表者名（部分一致含む） | ★★★ 最強 |
| **担当者経由** | 担当者名 ≠ 代表者名 | ★★ |
| **代表者情報なし** | 代表者フィールドが空 | ★ |
| **バイネームなし** | 担当者名が「担当者」または空 | - |

### 電話番号タイプ

| タイプ | 判定 |
|-------|------|
| 携帯あり | MobilePhoneに値あり |
| 携帯なし | MobilePhoneが空 |

### 最強セグメント

**「代表者直通 × 携帯あり」= 最も成約率の高いセグメント**

- 市来に優先割当
- 他メンバーへの均等分配には含めない
- 市来の合計件数は「最強セグメント件数 + 均等分配件数」

## 🔴 既知の問題と再発防止策

### 電話番号消失問題（2026-01-09発覚）

#### 問題の概要

新規リード作成時に電話番号が消失または破損する問題が発生。

| 問題 | 件数 | 症状 |
|-----|------|------|
| 先頭0消失 | 約150件 | `08032369313` → `8032369313.0` |
| 電話番号空欄 | 約20件 | Phone/MobilePhone両方空 |

#### 根本原因

**CSVファイル結合時に `dtype=str` を指定せずに読み込んだため、電話番号がfloat型として解釈された。**

```python
# NG: 電話番号がfloatになる
df = pd.read_csv('file.csv')
# MobilePhone: 8.032369e+09 (先頭0消失 + 指数表記)

# OK: 文字列として読み込み
df = pd.read_csv('file.csv', dtype=str)
# MobilePhone: '08032369313' (正しい形式)
```

#### 再発防止策

##### 1. CSV読み込み時のルール

**すべてのCSV読み込み時に `dtype=str` を指定すること**

```python
# 必須パターン
df = pd.read_csv('file.csv', dtype=str)
```

##### 2. CSV書き込み時のルール

**電話番号列は先頭にクォートを付けて出力すること**

```python
# 電話番号列にクォートプレフィックスを付与
df['Phone'] = df['Phone'].apply(lambda x: f"'{x}" if pd.notna(x) and x else x)
df['MobilePhone'] = df['MobilePhone'].apply(lambda x: f"'{x}" if pd.notna(x) and x else x)
```

または `quoting=csv.QUOTE_ALL` を使用：

```python
import csv
df.to_csv('file.csv', quoting=csv.QUOTE_ALL, index=False)
```

##### 3. インポート前チェック

**電話番号形式を必ず確認すること**

```python
# 電話番号形式チェック
def validate_phone(phone):
    if pd.isna(phone) or not phone:
        return True  # 空は許容
    phone = str(phone)
    # .0が含まれていないか
    if '.0' in phone:
        return False
    # 先頭が0か
    if not phone.startswith('0') and not phone.startswith("'"):
        return False
    return True
```

##### 4. チェックリスト追加項目

インポート前チェックに以下を追加：

- [ ] 電話番号に `.0` が含まれていないか
- [ ] 電話番号が10〜11桁か
- [ ] 電話番号が先頭0で始まっているか

### 成約先漏れ問題（2026-01-09発覚）

#### 問題の概要

電話番号突合では除外できない成約先が15件存在した。

#### 根本原因

**成約先の電話番号とスクレイピングデータの電話番号が異なる場合、電話番号突合では検出できない。**

例：医療法人財団日扇会第一病院
- Account.Phone: 03-3718-7281
- スクレイピング: 03-3718-7481（リハビリ問い合わせ用の別番号）

#### 再発防止策

##### 1. 会社名突合の追加

電話番号突合に加え、会社名の部分一致でも成約先をチェックする。

```python
def check_contract_by_name(company_name: str, contract_names: set) -> bool:
    \"\"\"会社名で成約先チェック\"\"\"
    normalized = company_name.replace(' ', '').replace('　', '')
    for suffix in ['株式会社', '有限会社', '医療法人', '社会福祉法人']:
        normalized = normalized.replace(suffix, '')

    for contract_name in contract_names:
        if len(contract_name) >= 4:
            if contract_name in normalized or normalized in contract_name:
                return True
    return False
```

##### 2. 成約先電話番号の拡張

Account.Phoneだけでなく、以下も除外対象に含める：
- Account.Phone2__c
- Account.PersonMobilePhone
- 関連Contact全員のPhone/MobilePhone

##### 3. インポート前の二重チェック

- [ ] 電話番号突合で除外
- [ ] **会社名突合で除外（新規追加）**
- [ ] 手動で疑わしい会社名をレビュー
